LADR Visualizer · Design language v0.1

How the visualizer should look, feel, and behave.

A short reference for the design language used in the LADR Visualizer prototype. This is the system, not the implementation: principles, tokens, layout patterns, and the few interaction conventions that, if held consistently, will make the product feel coherent across both its modes.

01 — Principles

Five commitments that everything else descends from. When in doubt, look here.

01.1

Be honest about representation.

The tool shows mathematical objects at the appropriate level of abstraction. Geometric where geometry helps; abstract diagrams, matrix heatmaps, or symbolic forms where it doesn't. Every visualization carries a small badge stating what kind of representation it is and whether its values are exact or approximate. Never pretend a 17-dimensional object is an arrow.

01.2

Sandbox, not slideshow.

The user defines objects, composes them, and views the results across multiple renderers. The UI exposes constructive operations directly, not a curated sequence of scenes. Even Browse mode is a way into the sandbox, not a destination.

01.3

Two modes, one room.

Browse and Sandbox share the same chrome, tokens, badges, and type pairings. The difference between them is density and emphasis, not personality. A user crossing from one to the other should never feel they have entered a different product.

01.4

Calm density.

The Sandbox is dense by necessity, but density is achieved through ruthless consistency — same spacing scale, same control sizes, same chrome — not by cramming. Whitespace earns its place even in a workbench.

01.5

Mathematics rendered as mathematics.

Variables, operators, and inline math are set in a true math face (STIX Two Text italic). UI labels, headings, and prose are set in a clean sans (Geist). The two are never mixed; the boundary between "this is interface" and "this is mathematics" is visible at a glance.

02 — Two modes, one system

Browse is the catalog of LADR concepts. Sandbox is the workbench. They are two views of the same system.

Browse mode presents the concepts of Linear Algebra Done Right as an editorial catalog. Wide margins, larger headings, generous whitespace, the page reads more like an extended table of contents than a tool. The unit is the concept card — a small, self-contained chip showing chapter reference, title, mini-visualization, teaser, and representation kind. Cards expand in place for a quick definition peek and open into a secondary side panel for the full treatment.

Sandbox mode is a fixed three-column workbench. Same tokens, tighter spacing. The user manipulates objects directly — spaces and operators are first-class items in a library on the left; visualizations of the current selection live as tiles in the center canvas; the inspector on the right shows computed properties of whichever object is focused. A math input bar sits at the bottom, always available.

The mode toggle lives in the top bar and is the only chrome that changes between them. Everything else — palette, badges, type, chrome, dialog conventions — is shared.

03 — Color & palette

Four palettes, one structure. Bone is the default. All four are interchangeable at the token level — no component encodes a hex value directly.

Bone — default
Ink-only accent. Most reserved.
Stone
Warm clay accent.
Slate
Cool graphite, blue accent.
Sage
Green-grey paper, forest accent.

Each palette exposes the same token surface — components do not need to know which palette is active. The roles are:

  • --bg, --bg-2, --bg-3 — page background, recessed surfaces, lowest tier
  • --panel, --panel-2 — raised surfaces (cards, tiles, panels) and their headers/feet
  • --ink, --ink-2, --ink-3, --ink-4 — text from primary to most muted
  • --line, --line-2, --line-3 — separators, light to heavy
  • --accent, --accent-soft, --accent-line — used only for selection state, the active item in a list, and the focus ring on inputs. Not for decoration.

Discipline: the accent never appears as a flat fill on a large surface, never gradients, and never carries emotional weight (no error reds, no success greens — those roles are filled by the kind badges, below). Saturation across the system is low; whites stay under 0.02 chroma. The product should feel like a serious paper, not a UI.

Rules

  • Headings, section titles, page titles, and concept-card titles use Geist. Never STIX italic.
  • Concept-card titles specifically use Geist Mono 500 — they read as technical labels, not as essay titles.
  • Geist Mono carries any metadata that belongs to the system rather than the math: chapter references, basis indicators, dimensions, timestamps, exactness chips, status text.
  • STIX Two Text italic is reserved for actual mathematical content: variable names, operator symbols, definitions, statements, expressions. Inside running prose, switch to STIX italic only for the math glyphs themselves.
  • Never use STIX Two Text for decoration or to suggest "this section is fancy." The italic carries information — it tells the reader they're looking at mathematics.

05 — The honesty system

Two small chips, applied consistently, that make the tool's epistemic stance visible.

Representation kind

Every visualization tile and every concept card declares what kind of representation it is. This is the heart of the "honest about representation" principle — the badge tells the user not to over-interpret what they're seeing.

geometric abstract diagram matrix spectral symbolic
  • Geometric — a literal picture of the object in ℝ² or ℝ³. Used only when the object actually lives there.
  • Abstract diagram — spaces as nodes, maps as arrows. Used for objects whose structure is relational (quotients, dual spaces, exact sequences) rather than spatial.
  • Matrix — a coordinate-dependent grid, usually rendered as a signed heatmap. Always disclosed as basis-dependent.
  • Spectral — eigenvalues or singular values plotted in ℝ or ℂ. The unit circle is a useful background reference (isometries land on it).
  • Symbolic — algebraic expressions, polynomial factorizations, dimensions. Always exact.

Exactness

exact approximate

Every value the system surfaces is one or the other. Exact values come from the symbolic computation layer (the computer-algebra side of the engine); approximate values come from numerical routines used for visualization. The distinction matters mathematically — a minimal polynomial computed symbolically over ℚ is not the same artifact as a characteristic polynomial fitted from floating-point eigenvalues. Surface it; do not blur it.

The card

Each card holds: chapter reference (Geist Mono, muted), title (Geist Mono, primary), a mini preview SVG rendered in the same representation it claims, a two-line teaser, the kind badge, and a prerequisite hint. Cards are pure information; no buttons compete with the click target.

Two ways to open a concept

  • Expand in place. A small "Read definition ⌄" chip in the card's bottom-right toggles an inline expansion. The teaser unclamps, and if the concept has authored formal content, the full STIX-typeset definition appears in a soft inset block. The card grows; neighbors reflow. Click again to collapse. This is the lightweight peek.
  • Open the side panel. Clicking the card body opens a fixed-position panel anchored to the right edge of the viewport. The catalog reflows: it loses its centered max-width and gains right padding equal to the panel's width. The grid stays visible and clickable. No dimming overlay, no modal — the panel is a sister column, not a dialog.

The side panel

The panel is the full treatment of a concept. It contains, in order: header (chapter, title, kind badge, exactness chip, prerequisites); a formal definition in a soft inset card with citation; the canonical visualization rendered larger, with its own kind badge and viz-foot metadata (basis, field); examples and non-examples in a two-column grid; related concepts as pill links; and an action row whose primary CTA is Open in Sandbox.

Sandbox — three columns plus persistent input bar. Layout is fixed; tiles flex.

Object library (left)

A list of every object the user has defined in the session, grouped by kind (Spaces, Operators & maps, Vectors, Inner products, …). Items show a glyph, a name in STIX italic (because they're math), and a type label in mono. Selection moves a single accent border to the chosen item; the inspector and the focused viz tile follow.

Canvas (center)

A grid of visualization tiles. Each tile is a self-contained unit: head with kind badge, math-typeset source label (e.g. 𝓜(T, ε), spec(T)), and tile actions; body with the visualization itself; foot with metadata (basis, field, exactness). Tiles are interchangeable through the visualization registry — any tile can be replaced with any other applicable view of the same object. The user composes their own bench; the system does not dictate a layout.

Inspector (right)

Properties of the focused object, organized in flat label-value pairs. Definition first (domain, codomain, field, basis), then computed properties (rank, nullity, det, trace, operator norm, condition number, structural flags like self-adjoint? with yes/no answers). Below that, a compact display of the minimal polynomial (or analogous symbolic invariant for the object), with provenance — via SymPy · exact. At the bottom, a list of "available visualizations" as pill links the user can drop into the canvas.

Math input bar (bottom)

A Symbolab-style two-row bar. Top row: a template palette of clickable insertions — fractions, powers, subscripts, roots, sums, integrals, matrix brackets, vector arrow, ⟨·,·⟩, ⊗, ⊕, ℝ, ℂ, λ, σ, ∈, ⊆, ∀. Bottom row: the live expression field, prefixed by a prompt glyph, with an Evaluate button on the right showing the return key affordance. The field renders math as math — italic STIX glyphs, real matrix brackets, real subscripts. It is not a text input that happens to contain math; it is a math input that happens to accept text.

08 — Visualization registry

The architectural pattern that lets the design hold.

A core commitment of the system: visualizations are not hard-coded to mathematical types. Instead, a runtime registry maps each mathematical type to the set of representations that can honestly render it. A linear map can be viewed as a matrix heatmap, an action on the unit cube, a spectral plot, an abstract diagram, an SVD ellipsoid, or a Jordan-form matrix — and the user picks. A quotient space cannot be viewed as a matrix heatmap; the registry simply does not offer that option.

This pattern is what allows the design to commit to honesty. The UI never has to make case-by-case decisions about whether a view makes sense — the registry has already filtered. Every tile in the canvas, every viz frame in the side panel, every "available visualizations" list in the inspector is reading from the same registry. New concepts and new visualizations plug in by registering; nothing in the chrome layer changes.

Two implications for the design:

  • Tile chrome is identical across visualization kinds. The kind badge is the only thing that signals what's inside. This consistency is load-bearing.
  • The user can swap any tile's visualization for another applicable one via the tile's overflow menu. The chrome stays put; the body changes.

09 — Cross-mode flow

How a user crosses from the editorial side to the working side without changing rooms.

The single most important interaction in the product. Browse mode's purpose, beyond being a catalog, is to seed the sandbox with concrete objects. Every concept's side panel has an Open in Sandbox action that does two things: switches the mode toggle to Sandbox and instantiates a fresh session pre-populated with the concept's canonical example (e.g. a concrete T : ℝ³ → ℝ³ for the eigenvalue concept), with the relevant visualizations already tiled in the canvas.

The reverse direction also exists, more quietly: every object in the sandbox library carries a small affordance that opens the corresponding catalog entry in the side panel without leaving the sandbox. The user gets the definition; the workbench keeps its state.

This is what makes the two-mode structure work as a single product. Browse is not "the marketing site." It is the doorway to the sandbox, and the sandbox owes its examples to the catalog.

10 — Do & don't

The fastest way to drift away from the system. Watch for these.

Do
  • Reach for a token before a hex value, always.
  • Set math in STIX Two Text italic, set everything else in Geist.
  • Show a kind badge on every visualization, every time.
  • Label exact vs approximate values wherever they surface.
  • Use the accent color sparingly — selection, active item, focus ring. That is the entire list.
  • Keep tile chrome identical across visualization kinds.
  • Let the side panel coexist with the catalog rather than overlay it.
Don't
  • Don't use STIX italic for headings, section titles, or decoration.
  • Don't fill large surfaces with the accent.
  • Don't introduce additional accent hues for "categories" — use the kind chip colors instead.
  • Don't render a high-dimensional or abstract object as an arrow because "the user expects a picture." Switch representations.
  • Don't open concepts as modal overlays. They are siblings of the catalog, not dialogs.
  • Don't surface basis-dependent values without saying so.
  • Don't introduce new gradients, glows, shadows beyond the existing 1-px borders and the panel's barely-there elevation.
